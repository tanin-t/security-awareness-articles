## เรื่องเล่าที่ทำให้คนทำ Tech ทั้งโลกอึ้ง

เดือนมีนาคม ปี 2019 มีข่าวที่ทำให้คนในวงการ tech อึ้งไปทั้งโลกครับ บริษัท **Facebook** ออกมายอมรับเองว่า ทีม security ภายในบังเอิญไปเจอเรื่องที่ไม่ควรเจอ — รหัสผ่านของผู้ใช้งาน **มากกว่า 600 ล้านบัญชี** ถูกเก็บไว้ในไฟล์บันทึกการทำงานของระบบ (log file) ในรูปแบบที่ **อ่านได้ตรงๆ ตัวๆ ไม่ได้ปกปิดอะไรเลย**

ลองนึกภาพนะครับ บัญชี Facebook ของคุณ ของแฟนคุณ ของคุณพ่อคุณแม่ — รหัสผ่านที่คุณคิดว่ามันเป็น "ความลับ" จริงๆ แล้วถูกพิมพ์ตรงๆ อยู่ในไฟล์ที่พนักงาน Facebook **มากกว่า 20,000 คน** เปิดดูได้ตลอดเวลา 7 ปี (ตั้งแต่ปี 2012)

Facebook ยืนยันว่า "ไม่พบหลักฐานว่ามีพนักงานนำไปใช้ในทางที่ผิด" — แต่ประเด็นไม่ใช่ตรงนั้นครับ ประเด็นคือ **มันไม่ควรอยู่ในสภาพที่ใครสักคนสามารถนำไปใช้ผิดได้ตั้งแต่แรก** ([อ่านข่าวต้นฉบับที่ KrebsOnSecurity](https://krebsonsecurity.com/2019/03/facebook-stored-hundreds-of-millions-of-user-passwords-in-plain-text-for-years/))

เรื่องคล้ายๆ กันยังเกิดกับบริษัทใหญ่อีกหลายเจ้า:

- **Twitter** ปี 2018 ออกมาบอกผู้ใช้ทุกคนทั่วโลกให้เปลี่ยนรหัสผ่าน เพราะมีบั๊กทำให้รหัสผ่านถูกเขียนลงในไฟล์ log ก่อนจะถูกแปลงเป็นรหัสลับ ([Twitter Blog](https://blog.twitter.com/en_us/topics/company/2018/keeping-your-account-secure))
- **GitHub** ปี 2018 ก็เจอปัญหาเดียวกัน — รหัสผ่านที่ user reset ถูก log เป็นข้อความธรรมดาโดยไม่ได้ตั้งใจ
- **Robinhood** (แอปลงทุน) ปี 2019 ก็มีรายงานว่ารหัสผ่านลูกค้าหลายพันคนถูก log เป็น plaintext

บริษัทระดับนี้ มีทีม security ระดับโลก มีงบประมาณไม่จำกัด **ยังพลาดได้** ครับ แสดงว่าเรื่อง plaintext password ไม่ใช่เรื่องที่ "แค่ระวังก็พอ" แต่ต้อง **ออกแบบให้ป้องกันไว้ตั้งแต่ต้น**

และในฝั่งคนใช้งานทั่วไปอย่างพวกเรา การส่งรหัสผ่านผ่านช่องทางที่ไม่ปลอดภัย เช่น chat, email, LINE — ก็คือการสร้าง "ไฟล์ log แบบ Facebook" ในเวอร์ชันส่วนตัวของเราเอง โดยไม่รู้ตัว

## Plaintext Password คืออะไร และทำไมถึงน่ากลัว

**Plaintext password** หรือ "รหัสผ่านในรูปแบบข้อความธรรมดา"[^1] คือรหัสผ่านที่อยู่ในรูปที่มนุษย์อ่านได้ทันที โดยไม่ผ่านการแปลงให้เป็นรหัสลับใดๆ เช่น:

- รหัสผ่านที่คุณพิมพ์ลงในช่อง login → ก่อนถูกระบบประมวลผล มันคือ plaintext
- รหัสผ่านที่คุณส่งให้เพื่อนผ่าน LINE → plaintext
- รหัสผ่านที่จดในไฟล์ Excel, แอป Notes, Google Doc → plaintext
- รหัสผ่านที่ developer เก็บใน database โดยไม่ได้แปลงเป็นรหัส → plaintext
- รหัสผ่านที่ปรากฏในไฟล์ log ของระบบ → plaintext

ปัญหาของการ "ปล่อยให้รหัสผ่านอยู่ในรูปอ่านได้" มี 3 ระดับครับ:

### ปัญหาที่ 1: ระบบถูกเจาะ = ทุกคนซวยพร้อมกัน

ถ้าฐานข้อมูลของบริษัทถูกเจาะเข้ามา **ผู้โจมตีจะได้รหัสผ่านของผู้ใช้ทุกคนทันที** ไม่ต้องถอดรหัส ไม่ต้องเดา ไม่ต้องประมวลผลอะไรเพิ่ม

แล้ว "ฐานข้อมูลถูกเจาะ" เกิดได้จากอะไรบ้าง? ลองดูตัวอย่างที่พบบ่อย:

- **SQL Injection**[^2] — เทคนิคที่ผู้โจมตีพิมพ์คำสั่งพิเศษลงในช่องกรอกข้อมูลของเว็บ (เช่น ช่อง search) เพื่อหลอกให้ระบบเผลอเปิดฐานข้อมูลให้ดู เปรียบเหมือนกับการที่คนร้ายพูดประโยคพิเศษกับยามหน้าตึกแล้วยามเปิดประตูให้เข้าโดยไม่ตั้งคำถาม
- **Cloud Misconfiguration**[^3] — ตั้งค่า cloud ผิด เช่น เผลอเปิดฐานข้อมูลให้คนทั่วอินเทอร์เน็ตเข้าถึงได้ เปรียบเหมือนสร้างตู้เซฟอย่างดี แต่ลืมล็อกแล้ววางไว้หน้าบ้าน
- **Insider Threat**[^4] — ภัยจากคนใน เช่น พนักงานที่ลาออกแล้ว access ยังไม่ถูกตัด หรือพนักงานที่นำข้อมูลออกไปขาย

ถ้ารหัสผ่านในฐานข้อมูลถูกแปลงให้อยู่ในรูป "รหัสลับ" ที่ถูกต้อง (เรียกว่า hash[^5]) ผู้โจมตีจะต้องใช้เวลามหาศาลในการแกะกลับมาเป็นรหัสจริง — บางครั้งใช้เวลาเป็นปีก็ยังแกะไม่ออก แต่ถ้าเป็น plaintext ก็จบเกมตั้งแต่วินาทีแรก

### ปัญหาที่ 2: คนในก็เห็น (และนั่นแหละน่ากลัว)

แม้ระบบจะไม่ถูกเจาะจากภายนอก แต่ยังมี **คนภายในบริษัท** ที่เข้าถึงระบบได้เป็นปกติอยู่แล้ว เช่น:

- ทีม developer, admin, หรือคนดูแลฐานข้อมูล
- ทีม support ที่อ่าน log เพื่อแก้ปัญหาให้ลูกค้า
- ผู้รับเหมาภายนอกที่ได้สิทธิ์เข้าถึงระบบชั่วคราว
- พนักงานเก่าที่ลาออกแล้ว แต่ access ยังไม่ถูกตัด

คนเหล่านี้ส่วนใหญ่ซื่อสัตย์ครับ — แต่หลักการ "ให้สิทธิ์เท่าที่จำเป็น"[^6] บอกว่าเราไม่ควรให้ใครเห็นข้อมูลที่ไม่จำเป็นต้องเห็น และ **รหัสผ่านของผู้ใช้ ไม่ควรเป็นข้อมูลที่ใครก็ตามเห็นได้** — แม้แต่ CEO หรือ CTO ของบริษัทเอง

### ปัญหาที่ 3: User ก็พลาดได้ (และพลาดบ่อยมาก)

ถ้าคุณส่งรหัสผ่าน WiFi บ้านให้เพื่อนผ่าน LINE คุณกำลังสร้างสำเนาของรหัสผ่านนั้นไว้อย่างน้อย **5-6 ที่** ครับ:

- เครื่องคุณ (chat history)
- เครื่องของเพื่อน (chat history)
- Server ของ LINE (เก็บไว้บน cloud)
- Backup ของทั้งสองฝั่ง (อาจจะอยู่ใน Google Drive, iCloud)
- Notification preview บนหน้าจอ lock screen ของผู้รับ
- บางครั้งมีคนเปิดมือถือดูข้ามไหล่โดยที่คุณไม่รู้

รหัสผ่านที่ส่งไปเพียงครั้งเดียว มี copy กระจายอยู่หลายที่ และคุณ **เอากลับมาไม่ได้** แม้จะลบ chat ก็ตาม

## เรื่องจริงที่กลายเป็นตำนาน

### RockYou (2009) — จุดเริ่มต้นของไฟล์ที่ hacker ทุกคนรู้จัก

บริษัท RockYou ทำเกมและแอปสำหรับโซเชียลมีเดียในยุคนั้น ในเดือนธันวาคม 2009 hacker ใช้ SQL Injection เจาะฐานข้อมูล แล้วพบว่าบริษัทเก็บรหัสผ่านของผู้ใช้ **32 ล้านบัญชี** เป็น plaintext ทั้งหมด — ไม่มีการเข้ารหัส ไม่มีการ hash อะไรทั้งสิ้น

ผลลัพธ์? ไฟล์ที่หลุดออกมาชื่อ "rockyou.txt" กลายเป็น **รายการรหัสผ่านที่ hacker ทุกคนทั่วโลกใช้เป็นเครื่องมือพื้นฐาน** จนถึงทุกวันนี้ ตั้งแต่นั้นมา ชื่อ "RockYou" กลายเป็นสัญลักษณ์ของความล้มเหลวด้านการเก็บรหัสผ่าน — และเป็นที่มาของชื่อ "RockYou2024" ที่เราพูดถึงในบทความที่แล้วด้วย ([อ่านเพิ่มเติม](https://www.imperva.com/blog/rockyou-32-million-passwords-revealed/))

### LinkedIn (2012) — ทำดีกว่าไม่ทำ แต่ทำผิดวิธีก็ยังเหมือนเดิม

LinkedIn ไม่ได้เก็บ plaintext แต่เก็บเป็น "รหัสลับ" ที่ใช้วิธีแปลงแบบเก่าและไม่ใส่เกลือ (salt[^7]) ซึ่งในยุคนั้นถือว่า "พอใช้" — แต่จริงๆ แล้ววิธีแปลงนั้นออกแบบมาให้คำนวณ **เร็ว** ซึ่งดีกับงานบางอย่าง แต่ **แย่มาก** สำหรับการเก็บรหัสผ่าน

เมื่อ hacker ได้ไฟล์รหัสลับมา **6.5 ล้านรายการ** ก็ใช้เวลาเพียงไม่กี่วันในการแกะรหัสผ่านส่วนใหญ่ออกมา ภายใน 72 ชั่วโมง สามารถแกะได้กว่า 90% ของรหัสผ่านทั้งหมด ([รายงานข่าว Ars Technica](https://arstechnica.com/information-technology/2016/05/117-million-linkedin-emails-and-passwords-from-2012-hack-up-for-sale/))

บทเรียนคือ: **ทำไม่ดี ก็เกือบเหมือนไม่ได้ทำ** ครับ

### Adobe (2013) — ทำผิดทุกอย่างที่ทำได้

Adobe โดนเจาะข้อมูล **153 ล้านบัญชี** สิ่งที่ทำผิดมีครบทุกข้อ:

1. ใช้วิธีเข้ารหัสที่ผิดประเภท (ใช้แบบ "เข้ารหัสได้-ถอดได้" แทนที่จะเป็นแบบ "แปลงทางเดียว" ซึ่งเหมาะกับ password มากกว่า)
2. ใช้โหมดการเข้ารหัสที่ทำให้รหัสผ่านที่เหมือนกัน ออกมาเป็นค่าเดียวกัน ทำให้ผู้โจมตีเดาได้ง่าย
3. **เก็บคำใบ้รหัสผ่าน (password hint) ไว้ในรูป plaintext** — เช่น "ชื่อสุนัขของฉัน" หรือ "ปีเกิด" → ผู้โจมตีไม่ต้องแกะรหัสด้วยซ้ำ แค่อ่านคำใบ้ก็เดาได้

ผลคือมีคนใจดี (?) สร้าง[เกมจิ๊กซอว์ครอสเวิร์ด](https://zed0.co.uk/crossword/)ที่ผู้เล่นช่วยกันเดารหัสผ่านได้ภายในไม่กี่วัน เป็นกรณีศึกษาคลาสสิกของ "ทำผิดทุกอย่างที่ทำได้" ([รายละเอียดเพิ่มที่ KrebsOnSecurity](https://krebsonsecurity.com/2013/11/adobe-breach-impacted-at-least-38-million-users/))

### Facebook, Twitter, GitHub, Robinhood — รหัสผ่านหลุดในไฟล์ log

ทั้งหมดนี้เก็บรหัสผ่านในระบบหลักได้ถูกต้อง แต่หลุดผ่านไฟล์ log โดยไม่ตั้งใจ — เป็นบั๊กที่คำสั่ง "บันทึกคำขอที่เข้ามา" เผลอบันทึกข้อมูลทั้งหมดที่ user ส่งเข้ามา (รวมถึง password) ก่อนที่จะถูกแปลงเป็นรหัส

บทเรียน: **บริษัทระดับโลกยังพลาดได้** ครับ — เพราะการเก็บรหัสผ่านให้ปลอดภัย ไม่ใช่แค่เรื่อง "เก็บใน database ยังไง" แต่รวมถึง "ระบบของเรา **ไม่เผลอ** เก็บรหัสผ่านในที่ที่ไม่ควรเก็บ"

## สำหรับ User ทั่วไป: หยุดส่งรหัสผ่านผ่านช่องทางที่ไม่ปลอดภัย

ส่วนนี้สำคัญที่สุดสำหรับคนใช้งานทั่วไปครับ เพราะการส่งรหัสผ่านเป็นเรื่องที่เกิดขึ้นทุกวันในชีวิตจริง — บอกรหัส WiFi ให้เพื่อน ส่งรหัสบัญชี Netflix ให้น้อง ส่งรหัส server ให้เพื่อนร่วมงาน

### ช่องทางที่ "ห้าม" ใช้ส่งรหัสผ่าน

- **LINE, Messenger, WhatsApp, Telegram** (แชทปกติ) — แม้จะมีการเข้ารหัสระหว่างทาง แต่ chat history ยังเก็บอยู่ทั้งสองฝั่ง + อยู่บน cloud backup
- **Email** — ผู้ดูแล mail server เห็นได้ + สำเนาอยู่ใน "Sent folder" ตลอดไป + อาจผ่าน server หลายตัวที่ไม่ใช่ของเรา
- **SMS** — ขึ้นที่หน้า lock screen ทันที + ผู้ให้บริการเครือข่ายอาจเก็บไว้
- **Slack, Microsoft Teams** — Admin ขององค์กรเห็น message ได้ทุกข้อความ
- **Google Doc, Notion, Excel ที่แชร์** — ใครก็ตามที่ได้ link หรือถูก add เข้ามาในอนาคต จะเห็นได้
- **Sticky note บนจอ หรือใต้ keyboard** — กรณีคลาสสิกที่ยังเกิดขึ้นทุกวันในออฟฟิศจริง

### "ก็แค่บัญชี Netflix นะ ไม่เป็นไรหรอก"

ปัญหาไม่ใช่บัญชี Netflix ครับ แต่เป็น 2 อย่าง:

1. **คุณใช้รหัสผ่านนั้นซ้ำกับที่อื่นไหม?** (กลับไปอ่านบทความที่ 2) ถ้าใช่ การส่งรหัส Netflix = การส่งรหัสของบัญชีอื่นไปด้วยพร้อมกัน
2. **คุณกำลังสร้างนิสัย** ที่จะส่งรหัสผ่านผ่านช่องทางเดียวกันในอนาคต — และวันที่คุณต้องส่งรหัสบัญชีธนาคารหรือรหัสที่ทำงาน คุณจะส่งโดยไม่ทันคิด เพราะมันกลายเป็นเรื่องปกติไปแล้ว

### วิธีส่งรหัสผ่านให้ปลอดภัย — เลือกได้ตามสถานการณ์

#### วิธีที่ 1: ใช้ "ลิงก์ใช้ครั้งเดียว" (One-Time Secret Link)

บริการอย่าง [Bitwarden Send](https://bitwarden.com/products/send/), [Password Pusher](https://pwpush.com), หรือ [One-Time Secret](https://onetimesecret.com) ทำงานแบบนี้ครับ:

1. คุณเข้าเว็บ ใส่รหัสผ่านที่อยากส่ง → ได้ link มา
2. ส่ง link นั้นให้ผู้รับผ่านช่องทางไหนก็ได้
3. ผู้รับเปิด link → เห็นรหัสผ่าน 1 ครั้ง
4. หลังจากนั้น **link หมดอายุทันที** — ใครเปิดดูซ้ำก็เห็นแค่ "expired"

ข้อดีคือ แม้ chat history จะหลุด ก็ได้แค่ link ที่ใช้ไปแล้ว ไม่ใช่รหัสผ่านจริง วิธีนี้ฟรี ไม่ต้องสมัคร เริ่มใช้ได้เลย

#### วิธีที่ 2: แชร์ผ่าน Password Manager

ถ้าทีมหรือบ้านใช้ Password Manager อยู่แล้ว (เช่น 1Password, Bitwarden) สามารถแชร์ credential ผ่านระบบได้โดยตรง — ผู้รับจะเห็นในแอปของตัวเอง ไม่ต้องส่งผ่าน chat เลย → จะมีรายละเอียดในบทความที่ 4

#### วิธีที่ 3: แบ่งช่องทาง — ส่งคนละทาง

ถ้าจำเป็นต้องส่งผ่าน chat จริงๆ ให้ใช้กลยุทธ์ **"ไข่ไม่อยู่ในตะกร้าใบเดียว"**:

- ชื่อผู้ใช้ส่งทาง email
- รหัสผ่านส่งทาง SMS หรือ **โทรบอกกันตรงๆ**
- หลักการคือ: ถ้าใครเจาะ email ของคุณได้ ก็ไม่ควรได้ข้อมูลครบชุด

วิธีนี้ไม่ดีเท่าวิธีที่ 1 แต่ก็ดีกว่าส่งครบในที่เดียว

#### วิธีที่ 4: ส่ง "วิธี reset" แทนรหัสผ่านเดิม

วิธีที่ปลอดภัยที่สุดคือ **ไม่ต้องส่งรหัสผ่านเลย** — ให้ผู้รับ reset password เองผ่านระบบ "forgot password" แล้วใช้รหัสผ่านใหม่ของตัวเอง คุณจะไม่เห็นรหัสของเขา และเขาก็ไม่ต้องเชื่อใจช่องทางการส่งของคุณ

## Action Items สำหรับทุกคน — ทำได้เลยวันนี้

1. **เลิกส่งรหัสผ่านผ่าน chat ตั้งแต่วันนี้** — ครั้งหน้าที่ต้องส่งให้เพื่อนหรือทีม ลองใช้ [onetimesecret.com](https://onetimesecret.com) (ฟรี ไม่ต้องสมัคร) หรือ [pwpush.com](https://pwpush.com) แทน
2. **ลบ chat history ที่มีรหัสผ่าน** — ลองไล่หา chat เก่าใน LINE/Messenger/Email ที่มีรหัสผ่านอยู่ → ลบทิ้ง + เปลี่ยนรหัสผ่านนั้นทันที (เพราะ "ลบ chat" ไม่ได้แปลว่ารหัสนั้นปลอดภัยอีกต่อไป)
3. **ตรวจ Notes / Excel / Google Docs** ที่เคยจดรหัสผ่านไว้ — ย้ายไปอยู่ใน password manager แทน → อ่านวิธีเริ่มต้นในบทความที่ 4
4. **ถามตัวเองทุกครั้งก่อนกดส่ง** — "ถ้าข้อความนี้หลุดออกไป จะเกิดอะไรขึ้น?" ถ้าคำตอบทำให้กังวล แสดงว่าควรเปลี่ยนช่องทาง

## บทเรียนชีวิตจากบทความนี้

> **ข้อมูลสำคัญต้องมี "ซองปิดผนึก" เสมอ — ไม่ว่าจะเป็นดิจิทัลหรือชีวิตจริง**

เวลาเราส่งเอกสารสำคัญทางไปรษณีย์ เราใส่ซอง ปิดผนึก บางครั้งใช้ EMS หรือลงทะเบียนเพื่อให้รู้ว่าใครเปิดเมื่อไหร่ — ไม่มีใครเขียนเลขบัตรประชาชนลงในโปสการ์ดแล้วฝากเพื่อนส่งให้

แต่ในโลกดิจิทัล เรากลับ "เขียนรหัสผ่านลงโปสการ์ด" ทุกวันโดยไม่รู้ตัว — ส่ง LINE บอกรหัส WiFi ส่ง email บอกรหัส server ทิ้งโน้ตที่มีรหัสไว้บนโต๊ะ เก็บรหัสผ่านลูกค้าไว้ใน Excel ที่แชร์กันในทีม

หลักการ "ซองปิดผนึก" ใช้ได้กับทุกเรื่องในชีวิตครับ:

- **เอกสารทางการ** — จดหมายสำคัญต้องลงทะเบียน
- **ความลับของคนอื่น** — เรื่องที่เพื่อนเล่าให้ฟัง ไม่ควรเอาไปเล่าต่อในที่สาธารณะ
- **ข้อมูลทางการเงิน** — เลขบัญชี รหัสบัตร ไม่ควรอยู่ในที่ที่คนทั่วไปเห็น
- **ความเห็นอ่อนไหว** — บาง message ที่จะส่ง ลองคิดว่าถ้าหลุดสาธารณะจะเป็นยังไง ก่อนกดส่ง

ความปลอดภัยที่แท้จริงไม่ได้มาจากการ "ระวัง" เพียงอย่างเดียว แต่มาจากการ **ใช้เครื่องมือและช่องทางที่ออกแบบมาเพื่อปกป้องข้อมูล** เหมือนเราไม่ได้แค่ "ระวังไม่ให้รถชน" แต่เราใช้เข็มขัดนิรภัย air bag และระบบเบรกที่ดี

ครั้งหน้าที่จะส่งอะไรสำคัญ ลองหยุดสัก 5 วินาทีแล้วถามตัวเองว่า: **"ถ้าข้อความนี้หลุดออกไป จะเกิดอะไรขึ้น?"** ถ้าคำตอบทำให้กังวล ก็แสดงว่าคุณกำลังเขียนลงโปสการ์ดอยู่

## ส่วนพิเศษสำหรับ Developer — Plaintext, Hashing, Encryption ใช้อะไรเมื่อไหร่

ส่วนนี้ลึกลงในเชิงเทคนิคสำหรับคนเขียนโค้ดโดยเฉพาะครับ ถ้าไม่ใช่ dev จะข้ามได้ แต่อยากให้ลองอ่านดูเพราะจะเข้าใจว่าทำไมบางบริษัทถึงพลาดเรื่องนี้

### Plaintext — ห้ามเก็บ ห้ามส่ง ทุกกรณี

ไม่มีกรณีไหนเลยที่ "ควร" เก็บรหัสผ่านเป็น plaintext ถ้าระบบของคุณยังเก็บอยู่ ให้ migrate ทันที — กลยุทธ์ที่ปลอดภัยคือ hash ตอน user login ครั้งถัดไป แล้วลบ plaintext ทิ้งทันทีหลังจากนั้น

### Encryption (เข้ารหัส) — สำหรับข้อมูลที่ต้องอ่านกลับ

**Encryption**[^8] เหมาะสำหรับข้อมูลที่ระบบ **ต้องอ่านกลับมาใช้ได้** เช่น:

- เลขบัตรเครดิต (ต้องส่งให้ payment gateway)
- เลขบัญชีธนาคาร (ต้องแสดงให้ user เห็น)
- API key ของ third-party ที่ระบบต้องเรียกใช้

Encryption ต้องใช้ **key** ในการเข้าและถอดรหัส ความปลอดภัยจึงขึ้นอยู่กับการเก็บ key ที่ดี → อ่านเพิ่มในบทความที่ 12 เรื่อง Secrets Management

**สำคัญ:** Encryption **ไม่เหมาะ** กับการเก็บรหัสผ่าน เพราะระบบไม่จำเป็นต้องรู้รหัสผ่านจริงของ user — แค่ต้องเปรียบเทียบว่าตรงกันไหม การใช้ encryption กับ password คือสิ่งที่ Adobe ทำผิดในกรณีศึกษาข้างต้น

### Hashing (แฮช) — สำหรับรหัสผ่านโดยเฉพาะ

**Hashing** คือการแปลงข้อมูลผ่านฟังก์ชันทางเดียว — แปลงเข้าได้ แต่ย้อนกลับไม่ได้ (หรือยากมากจนไม่คุ้มที่จะทำ) เหมาะกับรหัสผ่านเพราะ **ระบบไม่ต้องรู้รหัสผ่านจริง** มันแค่เปรียบเทียบว่า hash ของรหัสที่ user พิมพ์ ตรงกับ hash ที่เก็บไว้หรือไม่

แต่ไม่ใช่ทุก hash function จะเหมาะกับ password:

| Algorithm | เหมาะกับ password? | เหตุผล |
|---|---|---|
| MD5 | ❌ ไม่เลย | เร็วเกินไป + เจอ collision attack มานานแล้ว |
| SHA-1 | ❌ ไม่เลย | เร็วเกินไป + ถูกประกาศ deprecated แล้ว |
| SHA-256 | ⚠️ ไม่แนะนำ | ออกแบบมาให้เร็ว → crack ง่ายด้วย GPU |
| **bcrypt** | ✅ ใช้ได้ | ออกแบบมาให้ช้า, มี salt ในตัว, ปรับ cost ได้ |
| **scrypt** | ✅ ใช้ได้ | ใช้ memory เยอะ → ต้านทาน GPU/ASIC |
| **Argon2** | ✅✅ แนะนำ | Winner ของ Password Hashing Competition (2015), state of the art |

หลักการเลือก password hashing มี 3 ข้อ:

1. **Slow by design** — รหัสผ่านที่ user พิมพ์ครั้งเดียวควรใช้เวลา hash ประมาณ 100-300ms (user ไม่รู้สึก แต่ทำให้ผู้โจมตีที่ลอง brute force หลายล้านครั้งช้าลงมหาศาล)
2. **มี salt** — ค่าสุ่มที่ผสมกับรหัสผ่านก่อน hash ทำให้รหัสเดียวกันของ user 2 คน ได้ค่า hash ต่างกัน → ต้านทานการโจมตีด้วย rainbow table[^9]
3. **Adaptive cost** — ปรับความซับซ้อนได้ตามยุคสมัย เพราะ hardware เร็วขึ้นเรื่อยๆ ปีนี้ cost factor 12 อาจเพียงพอ ปีหน้าอาจต้องเพิ่มเป็น 14

### Code example: ทำให้ถูก vs ทำให้ผิด

```python
# ❌ ผิด — เก็บ plaintext
db.save(user_id, password)

# ❌ ผิด — MD5/SHA-256 ไม่มี salt
import hashlib
db.save(user_id, hashlib.sha256(password.encode()).hexdigest())

# ✅ ถูก — bcrypt
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))
db.save(user_id, hashed)

# ✅ ดีกว่า — Argon2
from argon2 import PasswordHasher
ph = PasswordHasher()
hashed = ph.hash(password)
db.save(user_id, hashed)
```

### กฎเหล็กของการ Log

ห้าม log ข้อมูลเหล่านี้ทุกกรณี (ทั้งใน application log, access log, error log):

- รหัสผ่าน (raw หรือ hash ก็ตาม)
- API keys, tokens, session IDs
- เลขบัตรเครดิต, CVV
- ข้อมูลส่วนบุคคล (PII) ที่ไม่จำเป็นต่อการ debug

วิธีที่ดีคือใช้ **logging middleware** ที่ filter sensitive fields อัตโนมัติ — กำหนด blacklist ของ field name ที่ห้าม log (`password`, `token`, `secret`, `authorization`, ฯลฯ) ทุก framework สมัยใหม่มี mechanism นี้ ใช้มันครับ ตั้งครั้งเดียวใช้ได้ตลอด

นี่คือบทเรียนจาก Facebook, Twitter, GitHub ที่ในที่สุดก็พลาดเรื่อง log — อย่ารอให้ตัวเองพลาดก่อนถึงจะตั้ง

### Action Items เพิ่มเติมสำหรับ Developer

5. **ตรวจ codebase ว่ามีจุดไหนเก็บหรือ log รหัสผ่านผิดวิธี** — search keyword: `password`, `pwd`, `secret` ใน log config และ database schema
6. **ถ้าใช้ MD5/SHA-1/SHA-256 สำหรับ hash password → migrate เป็น Argon2 หรือ bcrypt** — กลยุทธ์ทั่วไปคือ rehash ตอน user login ครั้งถัดไป
7. **ตั้ง logging middleware ให้ filter sensitive fields** ในทุก framework ที่ใช้
8. **เพิ่ม pre-commit hook ที่ scan secrets ใน code** — ใช้ `gitleaks` หรือ `trufflehog` (อ่านเพิ่มในบทความที่ 12)

## อภิธานศัพท์ (Glossary)

[^1]: **Plaintext (ข้อความธรรมดาที่อ่านได้)** — ข้อมูลในรูปแบบที่มนุษย์อ่านได้ทันที โดยไม่ผ่านการเข้ารหัส (encryption) หรือแฮช (hashing) ใดๆ เป็นรูปแบบที่อันตรายที่สุดในการเก็บข้อมูลที่อ่อนไหว เพราะถ้าฐานข้อมูลถูกเจาะ ผู้โจมตีจะเห็นข้อมูลทุกตัวทันที

[^2]: **SQL Injection** — เทคนิคการโจมตีที่ผู้โจมตีพิมพ์คำสั่งฐานข้อมูล (SQL) ลงในช่องกรอกข้อมูลของเว็บ เช่น ช่องค้นหาหรือช่อง login เพื่อหลอกให้ระบบเผลอเปิดข้อมูลที่ไม่ควรเปิด ปัญหาเกิดจากระบบไม่ตรวจสอบสิ่งที่ user พิมพ์เข้ามาให้ดีก่อนนำไปใช้ เป็นช่องโหว่ที่อยู่ในรายการ OWASP Top 10 มานานหลายปี → อ่านเพิ่มในบทความที่ 14

[^3]: **Cloud Misconfiguration** — การตั้งค่าระบบบน cloud ผิดพลาดจนเปิดช่องให้คนนอกเข้าถึงได้ ตัวอย่างที่พบบ่อย: เปิดให้ฐานข้อมูลเข้าถึงได้จากอินเทอร์เน็ตทั่วไป (โดยตั้งใจให้แค่เซิร์ฟเวอร์ของตัวเองเข้า) หรือเปิด S3 bucket ที่เก็บไฟล์เป็น public โดยไม่ตั้งใจ → อ่านเพิ่มในบทความที่ 16

[^4]: **Insider Threat (ภัยจากคนใน)** — ความเสี่ยงด้านความปลอดภัยที่มาจากคนภายในองค์กรที่มีสิทธิ์เข้าถึงระบบอยู่แล้ว ไม่ว่าจะเป็นพนักงานปัจจุบัน ผู้รับเหมา หรือพนักงานเก่าที่สิทธิ์ยังไม่ถูกตัด อาจเกิดจากความตั้งใจ (ขโมยข้อมูล) หรือไม่ตั้งใจ (เผลอเปิดเผยข้อมูล)

[^5]: **Hash (ค่าแฮช)** — ผลลัพธ์จากการนำข้อมูลผ่านฟังก์ชันคณิตศาสตร์ที่แปลงข้อมูลเป็นรหัสที่มีความยาวคงที่ ลักษณะสำคัญคือเป็นทางเดียว — ได้ค่า hash แล้วย้อนกลับเป็นข้อมูลเดิมไม่ได้ (หรือยากมาก) เช่น "password123" อาจถูกแปลงเป็น "ef92b778bafe771e..."

[^6]: **Principle of Least Privilege (POLP)** — หลักการให้สิทธิ์เท่าที่จำเป็นต่อการทำงาน ไม่มากกว่านั้น ใช้ได้ทั้งกับบัญชี user, application, service และ API key เป็นหลักการพื้นฐานของ security ที่ลด attack surface และจำกัดความเสียหายเมื่อมีการละเมิด

[^7]: **Salt (เกลือ)** — ค่าสุ่มที่ถูกผสมเข้ากับรหัสผ่านก่อนนำไป hash ทำให้รหัสผ่านที่เหมือนกันของ user 2 คนมี hash ที่ต่างกัน ป้องกันการโจมตีด้วย rainbow table และทำให้ผู้โจมตีต้อง crack รหัสผ่านทีละคน แทนที่จะ crack ทั้งฐานข้อมูลพร้อมกัน

[^8]: **Encryption (การเข้ารหัส)** — กระบวนการแปลงข้อมูลให้อยู่ในรูปแบบที่อ่านไม่ได้ โดยใช้ key (กุญแจ) สามารถถอดรหัสกลับมาเป็นข้อมูลเดิมได้ถ้ามี key ที่ถูกต้อง เหมาะกับข้อมูลที่ต้องอ่านกลับมาใช้ เช่น เลขบัตรเครดิต, ข้อความที่ส่ง — ต่างจาก hashing ที่ย้อนกลับไม่ได้ → อ่านเพิ่มในบทความที่ 6

[^9]: **Rainbow Table** — ตารางที่คำนวณ hash ของรหัสผ่านที่เป็นไปได้จำนวนมากไว้ล่วงหน้า ทำให้ผู้โจมตีสามารถค้นหาจาก hash → รหัสผ่านได้อย่างรวดเร็ว การใส่ salt ทำให้ rainbow table ใช้ไม่ได้ เพราะต้องสร้างตารางใหม่สำหรับแต่ละค่า salt
