## เรื่องเล่าที่คุณอาจจะเคยเป็น

ลองนึกถึงเช้าวันจันทร์ที่ยุ่งๆ ครับ คุณเปิดคอมจะเข้าระบบของบริษัท ใส่รหัสผ่านที่ใช้ประจำ — "รหัสผ่านไม่ถูกต้อง" ลองอีกครั้ง ผิดอีก ลองตัวที่คล้ายๆ กัน ผิดอีก สุดท้ายต้องกด **"ลืมรหัสผ่าน"** ตั้งใหม่ — เป็นรหัสที่ใกล้เคียงของเดิมแต่เปลี่ยนเลขท้ายจาก `2024` เป็น `2025`

ตกบ่ายวันเดียวกัน ลูกค้าโทรมาขอรหัสเข้าระบบที่คุณตั้งให้เขาไว้เมื่อหกเดือนก่อน — คุณเปิด Excel ที่ชื่อว่า `passwords_final_final_v3.xlsx` ที่อยู่ใน Desktop แล้วก็ค้นหา ไม่เจอ เปิด Notes บนมือถือ ไม่เจอ เปิด chat กับเพื่อนร่วมทีมเก่า — เจอแล้ว แต่เป็นรหัสที่ตั้งไว้ก่อนเปลี่ยน... สุดท้ายต้อง reset ใหม่อีกรอบ

ตกเย็น มีข่าวว่าเว็บที่คุณเคยสมัครเมื่อปี 2018 ถูกเจาะข้อมูล รหัสผ่านที่ใช้คือ `Tanin@2018` — ซึ่ง... อืม... คุณก็จำไม่ได้แล้วว่ามันเหมือนกับรหัสที่ใช้ใน Gmail ตอนนั้นรึเปล่า

ถ้าเรื่องเหล่านี้ฟังคุ้นๆ ไม่ใช่เพราะคุณเป็นคน "ขี้ลืม" หรือ "ไม่ระวัง" ครับ — เพราะ **สมองมนุษย์ไม่ได้ถูกออกแบบมาเพื่อจำรหัสผ่านสุ่มหลายสิบตัวที่ต่างกันหมด** เราเลยใช้ "ทางลัดของสมอง" — รหัสซ้ำ รหัสง่าย รหัสที่จดไว้ในที่ที่ไม่ปลอดภัย — ซึ่งคือต้นเหตุของบทความที่ 2 และ 3 ที่ผ่านมา

ทางออกของปัญหานี้มีอยู่จริงครับ และมันเรียกว่า **Password Manager**

## ปัญหาที่ใหญ่กว่าที่คิด — ทำไมการจำเองถึงพัง

จากบทความที่ 2 เราเห็นแล้วว่า "ห้ามใช้รหัสซ้ำ" และจากบทความที่ 3 เราเห็นแล้วว่า "ห้ามเก็บ plaintext ในที่ที่ไม่ปลอดภัย" รวมกันแล้วได้สมการที่เป็นไปไม่ได้สำหรับมนุษย์:

> **สมการที่เป็นไปไม่ได้:** จำรหัสผ่านที่สุ่ม ยาว และไม่ซ้ำกันเลย สำหรับบัญชีออนไลน์เฉลี่ย 80–100 บัญชี/คน

ลองคำนวณดูครับ ถ้ารหัสผ่านที่ปลอดภัยควรยาวอย่างน้อย 16 ตัวอักษร ผสม upper/lower/number/symbol — คุณต้องจำสตริงสุ่มประมาณ 1,600 ตัวอักษรในหัว เป็นไปไม่ได้ครับ

ผลที่เกิดขึ้นในชีวิตจริง:

1. **ใช้ pattern ซ้ำๆ** — `Site1!Tanin2024`, `Site2!Tanin2024` → ผู้โจมตีเดา pattern ออก credential stuffing ทำงานได้ทันที
2. **จดไว้ในที่ที่ไม่ปลอดภัย** — Excel, Notes, sticky note ใต้แป้นพิมพ์, chat กับตัวเอง
3. **กด "ลืมรหัส" บ่อยจนเป็นนิสัย** — ทำให้ email account กลายเป็น single point of failure ของทั้งชีวิต
4. **ใช้ browser ช่วยจำแบบไม่รู้กลไก** — บางครั้งปลอดภัย บางครั้งไม่ ขึ้นกับการ sync และ master password

ทั้งหมดนี้ไม่ใช่ "ความผิดของคน" — เป็นความผิดของระบบที่บังคับให้คนต้องทำสิ่งที่สมองทำไม่ได้

## Password Manager คืออะไร — แบบเข้าใจง่าย

**Password Manager** (โปรแกรมจัดการรหัสผ่าน) คือโปรแกรมที่ทำหน้าที่เป็น "ตู้เซฟรหัสผ่าน" ส่วนตัวของคุณ — เก็บรหัสผ่านทั้งหมดของทุกเว็บที่คุณใช้ไว้ในที่เดียว และคุณจำแค่ **รหัสเดียว** เพื่อเปิดตู้เซฟนั้น

หลักการทำงานสรุปสั้นๆ มี 3 ส่วน:

### 1. Vault — ตู้เซฟที่เก็บข้อมูล

**Vault** (ตู้เซฟ — ฐานข้อมูลที่เก็บรหัสผ่านและข้อมูลลับอื่นๆ ของ password manager) คือไฟล์ที่เก็บรหัสผ่านของคุณทั้งหมด อยู่ในรูปแบบที่ **เข้ารหัส (encrypted)** ตลอดเวลา — แม้ใครจะได้ไฟล์นี้ไปก็เปิดอ่านไม่ได้ถ้าไม่มี master password

การ "เข้ารหัส" คือกระบวนการแปลงข้อมูลให้อยู่ในรูปแบบที่อ่านไม่ได้ ต้องมี key ที่ถูกต้องถึงถอดกลับมาเป็นข้อมูลเดิมได้ — password manager ใช้มาตรฐาน AES-256 ซึ่งเป็นมาตรฐานเดียวกับที่ใช้ในระบบทหารและธนาคาร → อ่านเพิ่มในบทความที่ 6

ในชีวิตประจำวัน vault ของคุณจะถูก sync ผ่าน cloud ของผู้ให้บริการ (เช่น 1Password servers, Bitwarden servers) ทำให้คุณใช้ได้ทั้งบนคอม มือถือ tablet — แต่สิ่งที่ผู้ให้บริการเห็นคือ **ก้อนข้อมูลที่เข้ารหัสแล้ว** ไม่ใช่รหัสผ่านจริง

### 2. Master Password — กุญแจดอกเดียวที่คุณต้องจำ

**Master Password** (รหัสผ่านหลัก) คือรหัสผ่านเดียวที่คุณต้องจำให้ได้ในชีวิต — ใช้เปิด vault

หลักการสำคัญ:

- master password ของคุณ **ไม่เคยถูกส่งไปที่ server ของผู้ให้บริการ** (ในระบบที่ออกแบบดี)
- ผู้ให้บริการ **ไม่รู้ master password ของคุณ** — เขา reset ให้คุณไม่ได้ ถ้าคุณลืมก็คือลืม
- การถอดรหัส vault เกิดขึ้น **บนเครื่องของคุณ** เท่านั้น — เรียกว่าโมเดล **zero-knowledge** (สถาปัตยกรรมที่ผู้ให้บริการไม่มีความสามารถในการอ่านข้อมูลของผู้ใช้ — แม้แต่พนักงานหรือผู้ดูแลระบบของผู้ให้บริการก็เห็นเพียงข้อมูลที่เข้ารหัสแล้ว)

ฟังดูน่ากลัวที่ "ลืมแล้วลืมเลย" แต่นี่คือคุณสมบัติด้านความปลอดภัยที่สำคัญที่สุด — ถ้าผู้ให้บริการ reset ได้ แปลว่าเขามี backdoor ซึ่งหมายความว่า hacker ที่เจาะเข้า server ของเขาก็มี backdoor เช่นกัน

### 3. Browser Extension / App — ตัวช่วยกรอกอัตโนมัติ

ส่วนนี้คือสิ่งที่ทำให้ password manager **ใช้สะดวกกว่าจำเอง** จริงๆ:

- เปิดหน้า login ของเว็บ → extension ตรวจ URL → autofill รหัสที่ตรงกัน
- สมัครบัญชีใหม่ → กดปุ่มเดียว → password manager generate รหัสสุ่มยาว 20 ตัว → save อัตโนมัติ
- เปลี่ยนเครื่อง → log in ด้วย master password → ทุกอย่าง sync มาทันที

โบนัสที่หลายคนมองข้าม: **autofill ป้องกัน phishing** (การโจมตีที่หลอกให้เหยื่อกรอกข้อมูลในเว็บปลอมที่ดูเหมือนเว็บจริง — รายละเอียดในบทความที่ 7) ได้ด้วย เพราะ password manager จะ autofill เฉพาะ URL ที่ตรงกับที่บันทึกไว้ — ถ้าคุณคลิก link ปลอมที่ดูเหมือน `gmaii.com` (สังเกต `i` สองตัว) password manager จะ **ไม่ autofill** ให้ → เป็นสัญญาณเตือนว่ามีอะไรผิด

## คำถามที่ทุกคนถามก่อนเริ่มใช้

### "เก็บรหัสทุกอย่างไว้ที่เดียว ไม่อันตรายเหรอ?"

นี่คือคำถามที่ดีและถูกต้องที่จะถามครับ — แต่คำตอบทางคณิตศาสตร์ของความเสี่ยงนั้นชัดเจน

ลองเปรียบเทียบ 2 สถานการณ์:

| สถานการณ์ | ความเสี่ยงเมื่อเว็บใดเว็บหนึ่งโดน | ความเสี่ยงเมื่อ master password หลุด |
|---|---|---|
| ใช้รหัสซ้ำ 80 บัญชี | บัญชีที่ใช้รหัสนั้น **ทั้งหมด** หลุดทันที | — |
| ใช้ password manager | บัญชีนั้น**เพียงบัญชีเดียว**หลุด | ทุกบัญชีหลุด (แต่ master ไม่ค่อยหลุดถ้าตั้งดี + เปิด 2FA) |

ทุกบัญชี vs ทีละบัญชี — ความเสียหายต่างกันมหาศาล และ master password 1 ตัวที่คุณตั้งให้แข็งแรง + เปิด 2FA ปกป้องได้ง่ายกว่าการพยายามตั้งรหัสที่ไม่ซ้ำ 80 ตัวด้วยตัวเอง

### "ถ้าผู้ให้บริการโดนเจาะล่ะ?"

เกิดขึ้นแล้วครับ — ในเดือนสิงหาคม 2022 บริษัท **LastPass** ถูกเจาะข้อมูล vault ของลูกค้าหลุดออกไป เหตุการณ์นี้กลายเป็นบทเรียนสำคัญของวงการ ([รายละเอียดที่ LastPass](https://blog.lastpass.com/posts/2022/12/notice-of-recent-security-incident))

แต่สิ่งที่เกิดขึ้นต่อ **ขึ้นอยู่กับการออกแบบของผู้ให้บริการ**:

- ในโมเดล zero-knowledge ที่ดี → ผู้โจมตีได้แค่ก้อนข้อมูลที่เข้ารหัส ต้อง brute force master password ของแต่ละคนเอง — ถ้า master password แข็งแรง (16+ ตัวอักษร สุ่ม) จะใช้เวลาเป็นล้านปี
- ในกรณี LastPass พบว่ามีลูกค้าบางส่วนที่ใช้ master password อ่อน → ถูก crack สำเร็จและสูญเสียทรัพย์สิน (โดยเฉพาะ crypto wallet)

บทเรียนคือ: **ความปลอดภัยของ password manager ขึ้นอยู่กับ master password ของคุณ** — ตั้งให้แข็งแรง เปิด 2FA และเลือกผู้ให้บริการที่มี architecture ดี

### "ลืม master password แล้วทำยังไง?"

ในระบบ zero-knowledge ที่แท้จริง — **ลืมคือลืม** ไม่มีใคร reset ให้ได้

ทางแก้ที่ทุกคนต้องทำตั้งแต่วันแรก:

1. ตั้ง master password เป็น **passphrase** — เช่น `วันจันทร์ฝนตก-MotorcycleHelmet-7242` (ยาว จำง่าย เดายาก)
2. เก็บ **emergency kit** หรือ **recovery code** ที่ปลอดภัย — เขียนใส่กระดาษ ใส่ตู้เซฟที่บ้าน หรือฝากไว้กับคู่ชีวิต
3. ตั้ง **emergency contact / family sharing** ในผู้ให้บริการที่รองรับ — เผื่อเกิดเหตุไม่คาดฝัน

## เปรียบเทียบตัวเลือกหลัก — เลือกอะไรดี

ตลาดมีตัวเลือกเยอะ แต่สำหรับคนทั่วไปและทีม dev ในไทย ขอเน้น 3 ตัวที่เด่นในแต่ละด้าน

### 1Password — ประสบการณ์ใช้งานที่ดีที่สุด

**ข้อดี:**

- UX ดีที่สุดในตลาด — autofill แม่น UI สวย รองรับครบทุก platform
- Watchtower — ตรวจรหัสที่อ่อน ซ้ำ หรือเคยหลุด
- Travel Mode — ลบ vault บางส่วนออกจากเครื่องชั่วคราวเวลาเดินทาง
- รองรับ passkey, secure notes, file attachment
- Team plan ใช้งานง่าย เหมาะกับองค์กร

**ข้อเสีย:**

- เสียเงิน (ประมาณ $3/เดือนสำหรับ personal, $8/user/เดือนสำหรับ teams) — ไม่มี free tier ที่จริงจัง
- Vault อยู่บน server ของ 1Password เท่านั้น (ไม่ self-host)

**เหมาะกับ:** คนที่ต้องการ "ใช้แล้วลืมว่ามันมีอยู่" + ทีมที่ไม่อยากเสียเวลาตั้งค่า

### Bitwarden — Open source ที่ความสมดุลดีที่สุด

**ข้อดี:**

- **Open source** — ใครก็ตรวจ code ได้ ผ่าน security audit หลายครั้ง
- Free tier ใช้ได้จริง — รหัสไม่จำกัด, sync ทุกเครื่อง — เหมาะกับคนทั่วไปไม่ต้องเสียตังค์
- เสียตังค์เพิ่มถ้าอยาก: family sharing, 2FA แบบ hardware key, file attachment
- **Self-host ได้** ผ่าน Vaultwarden — เหมาะกับทีมที่ต้องการเก็บข้อมูลในประเทศ
- Bitwarden Send — ระบบส่ง one-time secret (เคยกล่าวถึงในบทความที่ 3)

**ข้อเสีย:**

- UX ดี แต่ไม่เนียนเท่า 1Password — โดยเฉพาะใน mobile app
- Browser extension รุ่นเก่าเคยมีปัญหาเรื่อง autofill บางหน้า (รุ่นปัจจุบันแก้แล้วเป็นส่วนใหญ่)

**เหมาะกับ:** คนทั่วไปที่ขึ้นต้นจาก free tier + ทีมที่อยาก self-host

### KeePass / KeePassXC — Local-first, ไม่ใช้ cloud

**ข้อดี:**

- **Open source** + **ฟรี 100%** ตลอดไป
- vault เป็นไฟล์ `.kdbx` ที่อยู่บน**เครื่องคุณเอง** — ไม่ต้องเชื่อใจผู้ให้บริการเลย
- อยากให้ sync ก็เอาไฟล์ไปวางไว้บน Dropbox, Google Drive, OneDrive, Syncthing เอง

**ข้อเสีย:**

- **ความรับผิดชอบเรื่อง backup, sync, security เป็นของคุณทั้งหมด** — ไม่มีใครคอยช่วย
- UX ค่อนข้างเทคนิค — อาจไม่เหมาะกับคนที่ไม่ใช่ dev
- Mobile experience ไม่เนียน ต้องใช้ third-party app (เช่น Keepass2Android, Strongbox)

**เหมาะกับ:** dev ที่ paranoid เรื่อง cloud + คนที่อยาก full control + ไม่กังวลเรื่อง setup

### สรุปการเลือก

- **ไม่อยากคิดมาก ใช้ได้เลย** → 1Password (เสียตังค์)
- **อยากของดีฟรี ขยายได้** → Bitwarden (ฟรี → จ่ายเมื่อจำเป็น)
- **ต้องการ full control ไม่พึ่ง cloud** → KeePassXC

ทีม Dev ในไทยส่วนใหญ่จะลงตัวที่ **Bitwarden** หรือ **1Password** — ทั้งสองตัวมี team plan ที่ใช้งานได้จริงและ audit logs ครบ

## Hands-on: เริ่มใช้วันนี้ใน 30 นาที

ขั้นตอนต่อไปนี้ใช้ได้กับ password manager ตัวใดก็ได้ ใช้ Bitwarden เป็นตัวอย่างเพราะเริ่มฟรี

### Step 1: สมัครและตั้ง Master Password (5 นาที)

1. ไปที่ [bitwarden.com](https://bitwarden.com) → Sign Up
2. ตั้ง master password — ทำตามกฎ:
   - **อย่างน้อย 16 ตัวอักษร** (ยิ่งยาวยิ่งดี)
   - ใช้ **passphrase** (4–6 คำสุ่มเชื่อมกัน) แทน password ที่จำยาก เช่น `แมวบนหลังคา-Helmet-Coffee-7242`
   - **อย่าใช้ master password ซ้ำกับที่อื่น**เด็ดขาด
3. **เขียน master password ลงกระดาษ** เก็บในตู้เซฟที่บ้าน (วันแรกเสี่ยงลืมที่สุด)

### Step 2: ติดตั้งบนทุกอุปกรณ์ (10 นาที)

- Browser extension: Chrome, Firefox, Edge, Safari
- Desktop app: macOS, Windows, Linux
- Mobile: iOS (เปิด Password AutoFill), Android

### Step 3: เปิด 2FA ให้ Password Manager เอง (5 นาที)

นี่คือ **สำคัญที่สุดและคนลืมบ่อยที่สุด** — รายละเอียด 2FA จะอยู่ในบทความที่ 5 แต่หลักง่ายๆ คือ:

- เปิด Settings → Two-step Login → เลือก Authenticator app (เช่น Authy, Aegis, Google Authenticator)
- บันทึก **recovery codes** เก็บไว้คู่กับ master password ในกระดาษ
- **ห้ามใช้ SMS** เป็น 2FA สำหรับ password manager — ไม่ปลอดภัยพอ

### Step 4: Migrate รหัสที่มีอยู่ (10 นาที + ทำต่อเรื่อยๆ)

ทำเป็น 2 รอบ:

**รอบที่ 1 — ขนของเข้าบ้านใหม่:**

- Import จาก browser ที่จำรหัสไว้ (Chrome, Safari, Firefox มีฟังก์ชัน export/import)
- Import จาก password manager เก่า (ถ้ามี)
- พิมพ์เพิ่มเอาเฉพาะบัญชีสำคัญ — Gmail, Facebook, Banking, GitHub, Slack

**รอบที่ 2 — เปลี่ยนรหัสใหม่ทีละบัญชี (ทำเรื่อยๆ ไม่ต้องรีบ):**

จัดลำดับความสำคัญ — เปลี่ยนรหัสกลุ่มเหล่านี้ก่อน:

1. **Email หลัก** (เพราะใช้ reset อย่างอื่น)
2. **Banking, Investment, Crypto wallet**
3. **Cloud (AWS, GCP, Azure), GitHub, GitLab**
4. **Social media หลัก** (Facebook, Twitter/X, LINE, IG)
5. ที่เหลือ — ทำตอนเข้าใช้งานครั้งหน้า

ทุกครั้งที่เปลี่ยน ให้กดปุ่ม "Generate" ของ password manager → ได้รหัสสุ่มยาว 20 ตัว → save → ใช้

### Step 5: ลบรหัสเก่าออกจากที่ที่ไม่ปลอดภัย

- Browser ที่จำรหัสไว้ — ถ้าใช้ password manager แล้ว ปิดฟังก์ชันจำรหัสของ browser
- Excel, Notes, Keep, Sticky note — ลบ + เปลี่ยนรหัสที่อยู่ในนั้นทันที (ตามบทความที่ 3)
- chat history — ลบ message ที่มีรหัสผ่าน (และเปลี่ยนรหัส)

## Action Items สำหรับทุกคน — ทำได้เลยวันนี้

1. **เลือกและสมัคร password manager** — Bitwarden (ฟรี) หรือ 1Password (ทดลอง 14 วัน) — เลือกได้ใน 5 นาที
2. **ตั้ง master password เป็น passphrase ยาว 16+ ตัวอักษร** — เขียนลงกระดาษเก็บที่บ้าน + ห้ามใช้ซ้ำกับที่อื่นเด็ดขาด
3. **เปิด 2FA ให้ password manager เป็นอย่างแรก** — ใช้ Authenticator app ไม่ใช้ SMS + บันทึก recovery codes
4. **Migrate รหัสที่มีอยู่** — Import จาก browser ทันที จากนั้นเปลี่ยนรหัสเก่าให้เป็นรหัสสุ่มที่ generate จาก password manager **5 บัญชีต่อสัปดาห์** (เริ่มจาก email หลัก, banking, GitHub, social media)

## บทเรียนชีวิตจากบทความนี้

> **ใช้ระบบช่วยจัดการสิ่งที่ซับซ้อน แทนที่จะพึ่งพาความจำ**

มีเรื่องที่นักจิตวิทยา cognitive science พูดมานานแล้วครับว่า สมองมนุษย์มี **working memory** จำกัดมาก — ประมาณ 4–7 ชิ้นข้อมูลในแต่ละช่วงเวลา และ long-term memory ก็ไม่ใช่ที่เก็บที่เชื่อถือได้สำหรับข้อมูลที่ "สุ่มและไม่มีความหมาย" (เช่น `xK7#mP9$qR2!`)

ทุกครั้งที่เราพยายามให้สมองทำสิ่งที่มันไม่ถนัด เราต้องเสียพลังงาน focus เสียเวลา และมีโอกาสพลาดสูง

หลักการ "ใช้ระบบ ไม่ใช้ความจำ" ใช้ได้กับทุกเรื่องในชีวิต:

- **เงิน** — บันทึกค่าใช้จ่ายในแอปแทนการพยายามจำ
- **งาน** — ใช้ task manager (Todoist, Things, Notion) แทนการพยายามจำว่าต้องทำอะไรบ้าง
- **นัดหมาย** — Calendar app + reminder แทนการพึ่งความจำ
- **ความรู้** — note taking (Obsidian, Notion, Apple Notes) แทนการพยายามจำทุกอย่างที่อ่านมา
- **รหัสผ่าน** — Password Manager แทนการพยายามจำเอง

คนที่ **ตั้งใจเลือกที่จะไม่พึ่งความจำ** ไม่ได้แปลว่าความจำไม่ดี — แต่เข้าใจว่าสมองของเขามีของดีกว่าที่จะใช้ — คิด สร้างสรรค์ ตัดสินใจ — ไม่ใช่ทำหน้าที่เป็นที่เก็บข้อมูลสุ่ม

Password manager ไม่ใช่ "ทางเลือกของคนขี้ลืม" — มันเป็น **ตัวขยายความสามารถ** (cognitive offloading) ที่ปล่อยให้สมองคุณไปทำอย่างอื่นที่สำคัญกว่า

ครั้งหน้าที่คุณรู้สึกว่า "ฉันต้องจำสิ่งนี้ให้ได้" ลองถามตัวเองว่า — มีระบบที่จำแทนได้ไหม? ถ้ามี ให้ระบบทำหน้าที่นั้น แล้วเอาสมองไปคิดสิ่งที่มีคุณค่ากว่า

## ส่วนพิเศษสำหรับ Developer — Team Sharing, Secret Manager, และ CLI

ส่วนนี้ลึกลงในเชิงเทคนิคสำหรับคนเขียนโค้ดและทีมที่ดูแล infrastructure โดยเฉพาะครับ ถ้าไม่ใช่ dev จะข้ามได้ — แต่ถ้าทีมคุณมี shared credential ก็อยากให้ลองอ่านดูเพราะส่งผลกับความปลอดภัยของทั้งทีม

### Team Sharing — แชร์รหัสในทีมแบบมืออาชีพ

ในที่ทำงานเรามักต้องแชร์ credential — รหัส server, API key ของ third-party, บัญชีสมัครบริการที่ทีมใช้ร่วมกัน

วิธีที่ผิด (แต่พบเห็นบ่อย):

- ส่งใน Slack channel
- เก็บใน Notion / Confluence ที่ทุกคน access ได้
- แชร์ Google Sheet ที่ใส่ password ดิบๆ
- ส่งผ่าน LINE / Email เวลาคนใหม่เข้าทีม

วิธีที่ถูก — ใช้ **Shared Vault / Organization** ของ password manager

ทั้ง 1Password และ Bitwarden รองรับ:

- **Shared Vault** หรือ **Collection** — กลุ่มของ credential ที่แชร์ให้คนในทีม
- **Permission control** — กำหนดได้ว่าใครเห็นเฉพาะบางกลุ่ม ใครแก้ได้ ใครลบได้
- **Audit log** — ดูได้ว่าใครเปิดดูรหัสไหนเมื่อไหร่
- **เข้าออกทีม** — เมื่อพนักงานออก เพิกถอน access ทันทีจากที่เดียว → กลายเป็นจุดควบคุม access ที่ตามรอยได้

หลักการตั้งค่า shared vault ที่ดี:

1. **แยก vault ตามหน้าที่** — `Engineering`, `DevOps`, `Marketing` แทนที่จะ shared vault เดียวให้ทุกคน
2. **Production secrets แยกจาก Staging** — และจำกัด access ตาม role
3. **Rotate รหัสเมื่อมีคนออก** — แม้จะเพิกถอน access ใน password manager แล้ว แต่คนที่เห็นไปก่อนหน้านี้ยังจำได้

### Password Manager ≠ Secret Manager

password manager ออกแบบมาสำหรับ "human secrets" — รหัสที่คนต้องเข้าไปดูและกรอกเอง ไม่เหมาะกับ "machine secrets" — credential ที่ application/service เรียกใช้แบบ runtime

| Use case | เครื่องมือที่เหมาะ |
|---|---|
| รหัส Gmail, GitHub UI, AWS console | Password Manager (1Password, Bitwarden) |
| Database password ที่ app ต่อ runtime | Secret Manager (AWS Secrets Manager, HashiCorp Vault, Doppler) |
| API key ที่ใช้ใน CI/CD | Secret Manager + CI secrets |
| `.env` ใน local dev | password manager CLI inject เข้า env vars |

หลีกเลี่ยงการเก็บ production database password ใน password manager — มันไม่ได้ออกแบบสำหรับ rotate อัตโนมัติ ไม่มี API ที่เหมาะกับ machine consumption และ access pattern ตามไม่ทันการ scale → อ่านเพิ่มในบทความที่ 12

### CLI: inject secrets เข้า dev environment โดยไม่ต้องเก็บ `.env` บน disk

แทนที่จะเก็บ `.env` ที่มี API key อยู่บนเครื่อง dev — ใช้ password manager CLI dereference ตอน runtime

```bash
# 1Password CLI — เก็บแค่ reference ใน .env, ค่าจริงไม่ได้อยู่บน disk
# .env file
DATABASE_URL="op://Engineering/db-staging/url"
STRIPE_KEY="op://Engineering/stripe/api-key"

# รัน app — op resolve reference → inject เข้า env vars
op run --env-file=.env -- npm start
```

```bash
# Bitwarden CLI — script-friendly
export DATABASE_URL=$(bw get password db-staging)
```

ข้อดี: ไม่มี secret อยู่บน disk, audit log ทุกการเรียกใช้, revoke ได้ทันทีที่คนออกจากทีม + ไม่หลุดเข้า git โดยบังเอิญ

### Self-hosting: Vaultwarden

ทีมที่ต้องเก็บ vault ในประเทศ (PDPA, regulated industries) เลือก **Vaultwarden** ได้ — Bitwarden server ที่เขียนใหม่ใน Rust ใช้ resource น้อย รัน Docker compose ขึ้นได้ใน 10 นาที และใช้กับ Bitwarden client ปกติได้ทั้งหมด

ข้อควรระวัง:

- ตั้ง backup ของ `db.sqlite3` + `attachments/` ทุกคืน — ลืม backup = ทีมหมดทาง recover
- เปิด HTTPS ผ่าน reverse proxy (Caddy / Traefik) — Bitwarden client ปฏิเสธ HTTP
- responsibility เรื่อง security update / monitoring เป็นของทีมเอง

### Action Items เพิ่มเติมสำหรับ Developer

5. **เสนอให้ทีมใช้ password manager เป็น standard** — สร้าง Organization และย้าย shared credential เข้าไป + แยก vault ตามหน้าที่
6. **Audit รหัสที่ทีมแชร์อยู่ตอนนี้** — ที่อยู่ใน Slack, Notion, Google Doc → ย้ายเข้า shared vault → ลบของเก่า + rotate รหัสที่หลุดออกไปก่อนแล้ว
7. **ใช้ `op run` หรือ `bw get` แทนการเก็บ `.env` บน disk** — ตั้งค่าให้ทีม dev ทุกคนใช้เป็นมาตรฐาน
8. **แยก machine secrets ออกจาก password manager** — รหัส production database / runtime API key ย้ายไปอยู่ใน secret manager (อ่านเพิ่มในบทความที่ 12)
