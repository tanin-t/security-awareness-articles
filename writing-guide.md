# Writing Guide — CyberSecurity Awareness Series

คู่มือสำหรับ AI agent ที่จะเขียนหรือแก้ไขบทความในซีรีส์นี้ในอนาคต

---

## 1. กลุ่มเป้าหมาย (Audience)

**คนทั่วไป + dev** — ไม่ใช่ dev อย่างเดียว

ทุกบทความต้องเข้าถึงได้สำหรับคนที่ไม่มีพื้นทาง technical ด้วย — แต่ก็ต้องมีเนื้อหาเชิงลึกพอสำหรับ dev อ่านแล้วได้ประโยชน์

**วิธีจัดการ 2 audience ในบทความเดียว:**

- เริ่มด้วย story / case study ที่ทุกคนเข้าใจได้
- อธิบายแนวคิดหลักด้วยภาษาคน + analogy ในชีวิตจริง
- ส่วน **"สำหรับ User ทั่วไป"** ก่อน ส่วน **"ส่วนพิเศษสำหรับ Developer"** ทีหลัง
- ส่วน dev ต้องเปิดด้วยประโยคแบบ "ส่วนนี้สำหรับคนเขียนโค้ดโดยเฉพาะ ถ้าไม่ใช่ dev จะข้ามได้" เพื่อให้คนทั่วไปไม่รู้สึกว่าตัวเองหลุดจากกลุ่มเป้าหมาย

---

## 2. โทน (Tone)

- **Semi-formal Thai** — ลงท้ายประโยคด้วย "ครับ" เป็นบางจุด แต่ไม่ทุกประโยค
- **เล่าเรื่อง > สอน** — ใช้ case study จริงนำ ไม่ใช่ทฤษฎีก่อน
- **มี analogy ในชีวิตจริงเสมอ** เช่น "เปรียบเหมือนตู้เซฟที่ลืมล็อก" "เหมือนเขียนรหัสลงโปสการ์ด"
- **ไม่ตำหนิ user** — สมมุติว่าคนอ่านอยากปลอดภัยอยู่แล้ว แค่ไม่รู้วิธี

---

## 3. ภาษา (Language)

- **ภาษาไทย ผสม English technical terms**
- ครั้งแรกที่เจอ technical term: เขียนแบบ `**คำไทย** (English term)` หรือ `**English term** (คำไทยอธิบาย)`
- ตัวอย่าง: `**ผู้โจมตี** (attacker)`, `**Plaintext password** (รหัสผ่านในรูปข้อความธรรมดา)`

---

## 5. การอธิบายศัพท์ (Term Definitions)

**ห้ามใช้ Glossary section ที่ท้ายบทความ** (หรือ footnote `[^N]`)

**ให้อธิบายศัพท์แบบ inline** ที่จุดที่คำนั้นปรากฏครั้งแรก เพราะ:

- คนทั่วไปอ่านแล้วเจอ footnote ต้องเลื่อนไปเลื่อนกลับ → เสียอรรถรส
- การอธิบายในบริบทช่วยให้เข้าใจดีกว่าคำนิยามแห้งๆ

**รูปแบบที่แนะนำ:**

```markdown
... ใช้วิธี **hashing** (การแปลงข้อมูลผ่านสูตรคณิตศาสตร์ทางเดียว — แปลงเข้าได้
แต่ย้อนกลับไม่ได้) ทำให้ผู้โจมตี ...
```

หรือถ้าศัพท์มี Thai equivalent ที่ดี:

```markdown
... หลักการ **"ให้สิทธิ์เท่าที่จำเป็น"** (Principle of Least Privilege) ...
```

## 6. ลิงก์ภายนอก (External Links)

- **ทุก URL ที่ใส่ในบทความ ต้อง verify ว่าใช้งานได้จริง** ก่อน commit (ใช้ WebFetch ตรวจ)
- มีลิงก์อ้างอิงข่าวจริงเสมอเมื่อพูดถึง breach หรือ incident
- เลี่ยงแหล่งที่มีโอกาส 404 สูง — บทความข่าวเก่าใน Imperva / Ars Technica เคยพังมาแล้ว
- แหล่งที่นิ่งและน่าเชื่อถือ: blognone (ไทย, ถ้ามี), TechCrunch, KrebsOnSecurity, Wikipedia, Wayback Machine,

## 7. ความปลอดภัยของไฟล์โค้ด (Secret Scanner Safety)

**ห้ามใช้ prefix รหัส API key แบบจริงในตัวอย่างโค้ด** — แม้จะเป็นแค่ตัวอย่างประกอบ

GitHub push protection จะ block push ถ้าเจอ shape ที่ตรงกับ:

- Stripe (`sk_live_...`, `sk_test_...`, `pk_...`)
- AWS (`AKIA...`)
- GitHub (`ghp_...`, `ghs_...`)
- OpenAI (`sk-...`)
- ฯลฯ

**ใช้ placeholder syntax แทน:**

```python
# ❌ จะโดน block
api_key = "sk_live_4eC39HqLyjWDarjtT1zdp7dc"

# ✅ ปลอดภัย
api_key = "<your-stripe-secret-key>"
api_key = "STRIPE_SECRET_KEY_HERE"
```

UPDATE: >>> ทำการปิด push protection แล้ว สามารถยกตัวอย่างคล้ายจริงได้ แต่ต้องไม่เอาในลักษณะที่มี \* อยู่ตรงกลาง

```
api_key = "sk_live_4eC39HqL*********************************szdp7dc"
```

## 9. Cross-references ระหว่างบทความ

- ใช้ `→ อ่านเพิ่มในบทความที่ X` เมื่อพูดถึง topic ที่ลึกกว่านี้ในบทความอื่น
- ดู outline ทั้ง 23 บทความใน [readme.md](readme.md)
- ทำบทความใหม่แล้ว ให้ทำการอัพเดตลิงค์ของบทความเก่า ๆ ที่ลิ้งมาด้วย

## 10. รูปแบบ Action Items

- เป็น numbered list
- ขึ้นต้นด้วยกริยา เช่น: "เลิกส่ง...", "ตรวจ...", "ตั้ง..."
- เน้น **ทำได้วันนี้** ไม่ใช่ "ในระยะยาวควร..."
- ใส่ link ที่ไปทำได้ทันที (เช่น onetimesecret.com)
- 4 ข้อสำหรับ user ทั่วไป + 4 ข้อเพิ่มสำหรับ dev (เลขต่อกัน 5-8)

## 11. Header / Metadata

```markdown
เขียนโดย <model> | บรรณาธิการ: Tanin T.

---
```
